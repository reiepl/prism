# Purpose

This document defines the logical data architecture and storage model for PRISM.

Rather than describing a specific database implementation, it establishes how information is classified, stored, generated, and managed throughout the application.

It serves as the foundation for all persistence, repositories, calculation engines, and future database implementations.

---

# Scope

This document defines:

* Data architecture
* Data ownership
* Core entities
* Entity relationships
* Identifier strategy
* Metadata standards
* Versioning strategy

This document does **not** define:

* IndexedDB implementation
* Dexie schema
* SQL structures
* Repository implementation
* Business calculations

---

# Data Design Principles

PRISM follows these principles when managing data.

## 1. Store Facts, Generate Knowledge

The database stores facts entered by the user.

Knowledge is generated by calculation engines.

Examples:

**Store**

* Birth Date
* Birth Time
* Birth Location
* Gender
* Relationship Type

**Generate**

* Four Pillars
* Ten Gods
* Hidden Stems
* Five Elements
* Useful God
* Tong Shu Analysis
* Compatibility Analysis

Facts are permanent.

Knowledge is reproducible.

---

## 2. User Owns the Data

All personal information belongs to the user.

No mandatory cloud storage.

No external dependencies.

---

## 3. Offline First

All primary data exists locally.

Internet access should never be required to retrieve user information.

---

## 4. Domain Driven

The database reflects real-world concepts rather than UI screens.

Examples include:

* Person
* Relationship
* Event
* Note

rather than

* Profile Page
* Dashboard
* Settings Screen

---

# Data Architecture

PRISM organises all information into two categories.

```text
                    PRISM DATA
                         │
        ┌────────────────┴────────────────┐
        │                                 │
        ▼                                 ▼
   Master Data                    Generated Data
(User Entered Facts)          (Engine Calculations)
```

This separation ensures that calculated knowledge can always be regenerated from user-owned information.

---

# Master Data

Master Data represents permanent information entered or maintained by the user.

It is considered the single source of truth.

Characteristics:

* User created
* Editable
* Persistent
* Long-term
* Never automatically overwritten

## Core Master Entities

### Person

Represents an individual.

Examples of information:

* Name
* Birth Information
* Gender
* Contact Information
* Notes

---

### Relationship

Represents a connection between two or more people.

Examples:

* Parent
* Child
* Friend
* Colleague
* Business Partner
* Client

---

### Event

Represents significant moments in a person's life.

Examples:

* Marriage
* Graduation
* Career Change
* Business Launch
* Relocation

---

### Note

User observations and journal entries.

---

### Tag

User-defined labels for organisation.

---

### Settings

Application preferences.

Examples:

* Theme
* Language
* Calendar Preferences
* Calculation Preferences

---

# Generated Data

Generated Data is produced automatically by PRISM.

It is always derived from Master Data.

Characteristics:

* Engine generated
* Deterministic
* Reproducible
* Versionable
* Disposable

Generated data should never replace Master Data.

## Core Generated Entities

### Blueprint

Generated from birth information.

Contains:

* Four Pillars
* Hidden Stems
* Ten Gods
* Five Elements
* Na Yin
* Twelve Growth Stages
* Shen Sha
* Useful God

---

### Relationship Analysis

Generated compatibility analysis between people.

---

### Tong Shu Analysis

Generated daily recommendations and activity analysis.

---

### Timeline Analysis

Generated correlations between events and Luck Pillars.

---

### AI Insights *(Future)*

Summaries generated by AI using existing facts and engine outputs.

AI insights should never become the source of truth.

---

# Entity Relationships

```text
Person
│
├── Relationships
│
├── Events
│
├── Notes
│
└── Blueprint
        │
        ├── Relationship Analysis
        ├── Timeline Analysis
        └── Tong Shu Analysis
```

Master Data drives Generated Data.

Generated Data never modifies Master Data.

---

# Data Flow

```text
User
   │
   ▼
Master Data
   │
   ▼
Repository
   │
   ▼
Calculation Engines
   │
   ▼
Generated Data
   │
   ▼
Presentation Layer
```

All calculations originate from Master Data.

---

# Identifier Strategy

Every entity should have a globally unique identifier (UUID).

Benefits include:

* Stable references
* Import and Export
* Future cloud synchronisation
* Reliable relationships
* Simplified migrations

Identifiers should never change once created.

---

# Metadata Standard

Every entity should include common metadata.

Recommended fields:

* id
* createdAt
* updatedAt
* version
* deleted (soft delete)
* source (optional)

This provides consistency across the application.

---

# Versioning Strategy

Schema evolution should be handled through controlled database migrations.

Whenever calculation engines are updated, Generated Data may be regenerated without affecting Master Data.

This allows PRISM to improve its analytical capabilities while preserving user-owned information.

---

# Future Expansion

The data architecture is designed to support future modules without structural redesign.

Potential additions include:

* Calendar Records
* Tong Shu Sessions
* AI Reports
* Attachments
* Knowledge References
* Research Notes
* Plugin Data
* Synchronisation Metadata

---

# Guiding Statement

PRISM treats data as a long-term asset.

User-entered facts are preserved as the permanent source of truth.

Knowledge is generated from those facts through explainable calculation engines and may evolve as PRISM's capabilities improve.

This separation between facts and knowledge enables PRISM to remain accurate, maintainable, and extensible throughout its lifetime.
